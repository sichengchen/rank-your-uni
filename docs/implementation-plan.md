# Rank Your Uni Implementation Plan

Date: 2026-06-13

## Goal

Build a local web app that lets a user compare two universities at a time and produces a final personalized ranking from the user's pairwise preferences.

## Verified Ranking Editions

- QS World University Rankings 2026
- U.S. News Best Global Universities 2025-2026
- Times Higher Education World University Rankings 2026
- Academic Ranking of World Universities 2025

The app will use the union of each publisher's latest global top 100 universities. Because the same university appears under slightly different names across publishers, ingestion needs a normalization layer rather than simple string equality.

## Proposed Stack

- VitePlus, React, TypeScript
- TanStack Router for the app route
- shadcn/ui components added via CLI
- Static JSON data for v1, generated by an ingestion script
- Local browser storage for comparison progress and finished rankings

This keeps the first version fast to build and deploy without introducing a backend before we need accounts, sharing, or persistent cross-device results.

## Data Contract

Generated file: `src/data/universities.json`

```ts
type RankingSource = "QS" | "US_NEWS" | "THE" | "ARWU";

type University = {
  id: string;
  name: string;
  aliases: string[];
  country?: string;
  ranks: Partial<Record<RankingSource, number | string>>;
  sourceUrls: Partial<Record<RankingSource, string>>;
};
```

Rank values allow strings because some sources publish ties or rank bands.

## Pairwise Ranking Algorithm

Use a lightweight active-ranking flow:

1. Seed universities by aggregate source strength, with unranked source positions treated as below the top 100.
2. Present one 1v1 match at a time.
3. Record the user's winner as a directed preference edge.
4. Maintain a score using an Elo-style update so the app can produce useful interim rankings quickly.
5. Prefer future comparisons that reduce uncertainty: close scores, few comparisons, and universities near the user's current top.
6. Let the user stop once confidence is good enough, while also supporting "keep comparing" for a stronger ranking.

This is more practical than asking the user to exhaustively compare the full union set, which would be thousands of comparisons.

## UI Scope

- First screen is the comparison workspace, not a marketing page.
- Two university panels with source-rank chips, country, and known aliases.
- Three comparison actions: choose left, choose right, skip / not enough info.
- Progress bar with comparisons completed and estimated confidence.
- Live personalized top list alongside or below the match.
- Controls to undo, reset, export JSON, and copy final ranking.
- Responsive layout for desktop and mobile.

## Implementation Steps

1. Scaffold the VitePlus React app.
2. Add shadcn/ui components through the CLI.
3. Add ranking ingestion scripts and normalized static data.
4. Build the pairwise comparison state machine and scoring helpers.
5. Build the 1v1 UI, ranking sidebar, progress states, and persistence.
6. Add tests for normalization, scoring, pair selection, undo, and completion behavior.
7. Run the app locally, verify with the in-app browser, then commit with conventional commit messages.

## Open Decisions

- Whether to include all universities in the union by default or provide a smaller "focus mode" that starts with the likely top 50 from the union.
- Whether the final ranking should be strictly personalized or shown next to the aggregate publisher ranking for context.
- Whether to fetch/update rankings only during development or expose an in-app "last refreshed" data label.
